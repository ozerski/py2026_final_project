### Итоговый проект: Система мониторинга задач (Task Tracker) в рамках работы с _NRFS_ по услугам ШПД сегмента B2C. 
#### py_adv_student79

**Цель:** Создание *консольного приложения (CLI)* для мониторинга и контроля 
жизненного цикла проектных задач (решения 
проблем с доступом NRFS (*Not Ready For Sale*) в МКД).

**Функциональные требования:**
* Обработка отчётов из СУС Гермес в различных форматах (XLSX, CSV).
* Ведение статистики по задачам с основными атрибутами проблем NRFS, 
  которые содержатся в отчёте СУС Гермес "Отчёт по проблемам NRFS", в т.ч.: 
   - номер проблемы / адресной заявки / выданной в работу заявки NRFS,
   - адрес / GID ОРПОН,
   - охват и диапазон ДХ с признаком NRFS
   - статус (текущий этап в СУС Гермес), 
   - необходимый бюджет затрат (всего и на оборудование в отдельности), 
   - исполнитель, 
   - контрольная дата технической готовности (КД ТГ).
* Контроль состояний задач (переходов между статусами) и формирование 
  отчета "Воронка задач" с расчетом конверсии между этапами и сроками 
  устранения NRFS.
* Отслеживание на этапе "Формирование инсталлплатежа и согласование КД 
  СМР" наличия проблем на том же адресе для их включения в одну адресную 
  заявку.
* Генерация еженедельного отчёта о динамике устранения NRFS с использованием 
  условного форматирования по критериям отклонений от значений прошлой недели и
  КД ТГ.


**Результат:** Актуальная информация о статусах проблем NRFS и отчет по 
  устранения NRFS / использованию выделенного бюджета 
  на устранение устранения NRFS.

#### 1. Технологический стек

* **Язык:** Python 3.12+.
* **Управление зависимостями:** uv.
* **База данных:** SQLite или PostgreSQL (на выбор) - пока не используется.
* **ORM:** SQLAlchemy 2.0 (современный декларативный синтаксис `Mapped`, `mapped_column`) - пока не используется.
* **Работа с таблицами и форматированием:** `pandas`, `openpyxl`.
* **CLI-интерфейс:** `Click` или `Typer` (рекомендуется).
* **Тестирование:** `pytest`.

#### 2. Архитектура базы данных (Реляционная модель)  *- пока не используется*.

В базе данных должны быть реализованы следующие сущности:

1. **User (Исполнитель):** ID, Имя/ФИО, Роль (например, Разработчик, Тестировщик, Менеджер).
2. **Task (Задача):** ID, Заголовок, Описание, `assignee_id` (FK, может быть NULL, если задача в бэклоге не распределена), Статус (хранится как Enum или строка), Приоритет (числовой вес или Enum), Дедлайн (DateTime), Дата создания.
3. **TaskLog (История переходов):** ID, `task_id` (FK), Предыдущий статус, Новый статус, Дата изменения. (Опционально, но рекомендуется для сбора точного таймшита).


#### 3. Функциональные требования и модули

**Модуль 1: Парсинг и импорт данных (разные форматы)**\
Скрипт должен рекурсивно обходить указанную директорию (например, `.
/sources`) и обрабатывать файлы в зависимости от их расширения (XLSX, ~~ODS,~~
CSV).

**Модуль 2: Нормализация и дедупликация**\
**Очистка:** 
* Удаление в данных не значащей информации 
*  - поле "Адрес": "Пензенская область, " и т.д., замена полного типа 
     населёного пункта на краткий аналог.
* - поле Этап в Гермес: сокращение наименования.
* - поле "Диапазон ДХ с ограничением": указание первой и 
    последней квартиры из диапазона вместо полного списка.
* - удаление идентичных строк с одинаковой датой.

**Модуль 3: Аналитический отчет «Воронка проблем NRFS»**
* Формирование аналитики на основе текущего распределения проблем по этапам и 
  истории их изменений.
* **Расчет конверсии:**
  * *Общая конверсия:* Процент проблем NRFS / ДХ, 
  дошедших до фактического устранения от общего числа созданных проблем / ДХ.
  * *Промежуточная конверсия:* Процент перехода проблем с предыдущего 
   этапа на текущий (показывает «узкие места» в отработке этапов).
* **Визуализация:** Реализация воронкообразной или столбчатой диаграммы (Bar Chart) внутри итогового файла Excel средствами Python , отображающая количество сделок на каждом из этапов.

**Модуль 4: Экспорт базы данных**
* Реализация выгрузки таблиц из БД во внешние файлы.
* **Поддержка двух форматов:** XLSX (Excel) и ODS (OpenDocument Spreadsheet).
* Экспортироваться единый файл, где каждый этап вынесен на отдельную вкладку.


**Модуль 5: Генерация таймшита (Отчетность)**

* **Условное форматирование (Conditional Formatting):** Средствами Python (через `openpyxl`) в Excel-файл должны быть зашиты правила подсветки:
* Если `КД ТГ < Текущая дата` — строка или ячейка дедлайна заливается 
  **красным** цветом (Просрочено).
* Если `Фактическая дата устранения`  `< КД ТГ` — заливается 
  **зеленым** (Выполнено в срок).
* Если КД ТГ наступает в ближайшие 24 часа — **желтым** (Риск просрочки).

**Модуль 6: CLI-интерфейс**
Управление системой поддерживает использование системы команд и флагов в 
терминале.

Пример команд:
* `python main.py update --mode "new"`
* `python main.py update --mode "all"`
* `python main.py b2c_stage --deal-id 5 --stage-id 3`
* `python main.py report_nrfs_fixed --output --start_date "__" --end_date "__"`
* `python main.py report_nrfs_funnel --output "funnel_report.xlsx"`
* `python main.py report_nrfs_full --output "full_report.xlsx"`
* `python main.py export_db --format xlsx --output "backup.xlsx"`
* `python main.py export_db --format ods --output "backup.ods"`


#### 4. Требования к качеству кода

1. **Архитектура:** Четкое разделение ответственности.
2. **Типизация:** Использование `Enum` для строгих ограничений статусов и приоритетов, проверка статической типизации через `mypy`.
3. **Обработка ошибок:** Кастомные классы исключений (например, `InvalidStateTransitionError`, `UnassignedTaskError`), которые перехватываются на уровне CLI и трансформируются в человекочитаемый вывод.
4. **Тестирование:** опциональное покрытие юнит-тестами (`pytest`) конечного автомата переходов (попытки валидных и невалидных смен статусов) и логики сравнения приоритетов.
